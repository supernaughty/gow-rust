From build.bat (verified):
- `build.bat installer x64` runs the full pipeline: cargo build release x64 → stage binaries (excluding gow-probe.exe) → heat.exe → fix-guids.ps1 → candle.exe → light.exe
- `build.bat installer x86` runs the same pipeline for i686-pc-windows-msvc target

- Version is read from Cargo.toml by build.bat (PowerShell `Select-String` via temp file)
- fix-guids.ps1 is called with `-ExecutionPolicy Bypass` — already in build.bat, no CI change needed
- build.bat must be invoked with `shell: cmd` (it is a .bat file)

From 07-RESEARCH.md (verified):
- WiX 3.14.1.8722 is pre-installed on windows-latest but NOT on PATH
- Add to PATH via: `echo "${WIX}bin" >> $GITHUB_PATH` (bash shell step)
- This must be the FIRST step after checkout, before any build.bat call
- `$WIX` env var is set on windows-latest (path is version-agnostic)
- i686 build requires `ilammy/msvc-dev-cmd@v1` with `arch: amd64_x86` BEFORE cargo build
  (cc-rs invokes the C compiler to build liblzma from source; without this, linker gets architecture mismatch LINK1112)
- `dtolnay/rust-toolchain@stable` with `targets: i686-pc-windows-msvc` needed for x86 job
- Swatinem/rust-cache@v2 keys: `x64-release` and `x86-release` — different keys prevent cache collisions
- publish job can run on `ubuntu-latest` (cheaper; download-artifact + release upload need no Windows)
- `softprops/action-gh-release@v2` (NOT v3 — requires Node 24; v2 is last Node 20 line)
- `permissions: contents: write` required at workflow level for softprops action
- `actions/upload-artifact@v4` and `actions/download-artifact@v4` (NOT v3 — deprecated)

  <action>
Write `.github/workflows/release.yml` with the following exact content:

```yaml
name: Release
on:
  push:
    tags:
      - "v*"

permissions:
  contents: write

jobs:
  build-x64:
    name: Build x64 MSI
    runs-on: windows-latest
    steps:
      - uses: actions/checkout@v4

      - name: Add WiX Toolset to PATH
        shell: bash
        run: echo "${WIX}bin" >> $GITHUB_PATH

      - uses: dtolnay/rust-toolchain@stable

      - uses: Swatinem/rust-cache@v2
        with:
          key: x64-release

      - name: Build x64 MSI
        shell: cmd
        run: build.bat installer x64

      - name: Upload x64 MSI artifact
        uses: actions/upload-artifact@v4
        with:
          name: msi-x64
          path: target/gow-rust-*-installer-x64.msi
          if-no-files-found: error

  build-x86:
    name: Build x86 MSI
    runs-on: windows-latest
    steps:
      - uses: actions/checkout@v4

      - name: Add WiX Toolset to PATH
        shell: bash
        run: echo "${WIX}bin" >> $GITHUB_PATH

      - uses: dtolnay/rust-toolchain@stable
        with:
          targets: i686-pc-windows-msvc

      - name: Set up MSVC 32-bit environment
        uses: ilammy/msvc-dev-cmd@v1
        with:
          arch: amd64_x86

      - uses: Swatinem/rust-cache@v2
        with:
          key: x86-release

      - name: Build x86 MSI
        shell: cmd
        run: build.bat installer x86

      - name: Upload x86 MSI artifact
        uses: actions/upload-artifact@v4
        with:
          name: msi-x86
          path: target/gow-rust-*-installer-x86.msi
          if-no-files-found: error

  publish:
    name: Create GitHub Release
    needs: [build-x64, build-x86]
    runs-on: ubuntu-latest
    steps:
      - name: Download all MSI artifacts
        uses: actions/download-artifact@v4
        with:
          pattern: msi-*
          merge-multiple: true
          path: dist/

      - name: Create GitHub Release and upload MSIs
        uses: softprops/action-gh-release@v2
        with:
          files: dist/*.msi
          fail_on_unmatched_files: true
```

Critical ordering in build-x86 job (per research Pitfall 4 + Pitfall 5):
1. `actions/checkout@v4` first
2. `Add WiX Toolset to PATH` second (before any build)
3. `dtolnay/rust-toolchain@stable` with `targets: i686-pc-windows-msvc` third
4. `ilammy/msvc-dev-cmd@v1` with `arch: amd64_x86` FOURTH — must come AFTER rust-toolchain and BEFORE the build step. This sets the MSVC 32-bit environment so cc-rs uses the 32-bit C compiler when compiling liblzma-sys from source.
5. `Swatinem/rust-cache@v2` with `key: x86-release` fifth
6. `build.bat installer x86` with `shell: cmd` sixth

`permissions: contents: write` is at the workflow level (not just the publish job) — this is fine and avoids redundancy. The CI workflow (07-02-PLAN.md) has no permissions block, so it gets the default read-only token.

Do NOT use `shell: pwsh` for the build step — build.bat is a Windows batch file and must run under `shell: cmd`.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Tag push → release workflow | Only maintainers with push access can create v* tags; fork PRs cannot trigger this |
| GitHub Actions runner → artifact storage | MSI files stored as GitHub artifacts between jobs; no secrets embedded in MSIs |
| publish job → GitHub Release API | Uses GITHUB_TOKEN with `permissions: contents: write`; scoped to release creation only |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-03-01 | Tampering | Supply chain: softprops/action-gh-release@v2 | mitigate | Pin to `@v2` (not `@master`); v2 is a maintained stable line; monitor for security advisories |
| T-07-03-02 | Tampering | Supply chain: ilammy/msvc-dev-cmd@v1 | mitigate | Pin to `@v1` tag; widely used (1M+ downloads); re-evaluate if compromise is reported |
| T-07-03-03 | Elevation of Privilege | permissions: contents: write on workflow | accept | Scoped to release creation; GITHUB_TOKEN cannot access other repos or org secrets; least privilege for the task |
| T-07-03-04 | Information Disclosure | MSI artifact contents | accept | MSIs contain compiled Rust binaries only; no secrets, credentials, or private keys embedded |
| T-07-03-05 | Tampering | Tag spoofing triggers unauthorized release | accept | Only repo maintainers with push access can push v* tags; standard GitHub access control |
| T-07-03-06 | Denial of Service | build-x86 liblzma C compilation failure | accept | Pitfall 4 mitigated by correct ilammy/msvc-dev-cmd ordering; fallback is to rebuild x64-only at v0.2.0 if A4 assumption proves wrong |
</threat_model>
